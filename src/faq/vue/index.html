# 1. 组件通讯方式

  1. props/$emit/.sync + $emit('update: attr', val)
  2. $emit/$on：创建一个vue实例，通过一个空的vue实例作为桥梁实现vue组件间的通信，实现非父子组件通信的一种解决方案，所有的组件都可以上下平行地通知其他组件
//utils/event-bus.js
import Vue from "vue";
export default new Vue();
//main.js
import bus from "@/utils/event-bus";
Vue.prototype.$bus = bus;
//Parent.vue 
import Child1 from "./child1";
import Child2 from "./child2";
// child1.vue
mounted() {
  this.$bus.$on("getMsg1", res => {
    this.msgList.push(res);
  });
},
methods: {
  clickSend() {
    this.$bus.$emit("getMsg2", "hello from1:" + parseInt(Math.random() * 20));
  }
}
//child2.vue
mounted() {
  this.$bus.$on("getMsg2", res => {
    this.msgList.push(res);
  });
},
methods: {
  clickSend() {
    this.$bus.$emit("getMsg1", "hello from2:" + parseInt(Math.random() * 20));
  }
}

vue是单页应用，如果你在某一个页面刷新了之后，与之相关的EventBus会被移除，这样就导致业务走不下去。还要就是如果业务有反复操作的页面，EventBus在监听的时候就会触发很多次，也是一个非常大的隐患。这时候我们就需要好好处理EventBus在项目中的关系。通常会用到，在页面或组件销毁时，同时移除EventBus事件监听。

export default{
  destroyed(){
      $EventBus.$off('event-name')
  }
}

适合逻辑并不复杂的小页面，逻辑复杂时还是建议使用vuex

  1. provide/inject不论子组件嵌套有多深，都能获取到,非响应的,注入的值是组件初始化的值
  2. $parent/$children
  3. $refs如果在普通的DOM元素上使用，获取到的就是DOM元素；如果用在子组件上，获取的就是组件的实例对象。
  4. $attrs/$listeners,
    1. 当我们在组件上赋予一个非prop声明时，编译之后的代码会把这个属性当成原始属性对待，添加到html原生标签上，inheritAttrs: false将它去掉

# 组件传值常见使用场景分为以下三类:

  1. 父子组件通信: props; $parent/$children; provide/inject; $ref; $attrs/$listeners
  2. 兄弟组件通信: EventBus; Vuex
  3. 跨级通信: EventBus; Vuex; provide/inject; $attrs/$listeners

# vuex工作流程

vuex有五种属性，分别是State、 Getter、Mutation 、Action、Module
* state => 基本数据(数据源存放地)
* getters => 从基本数据派生出来的数据， 相当于state的计算属性
* mutations => commit提交更改数据的方法，同步操作！
* actions => dispatch 像一个装饰器，包裹mutations，使之可以异步。
* modules => 模块化Vuex

# Vue单页应用与多页应用的区别

  1. 定义
    1. 单页面应用：只有一个主页面的应用，一开始只需要加载一次css、js 等相关资源，所以内容都包含在主页面，对每个功能模块组件化，但页面应用跳转是切换相关组件，仅仅刷新局部资源。
    2. 多页面应用：有多个独立页面的应用，每个页面都必须重复加载js、css等相关资源。页面跳转需要整个页面刷新。
  2. 区别
    1. 刷新方式
      • SPA：相关组件切换，页面局部刷新
      • MPA：整页刷新
    2. 路由模式
      • SPA：可以使用hash, 也可以使用history
      • MPA：普通链接跳转
    3. 用户体验
      • SPA：用户体验良好，切换页面只是局部刷新组件，因为一次性加载资源，首次加载可能会出现白屏幕。
      • MPA：页面切换缓慢，流畅度不够，用户体验比较差
    4. 转场动画
      • SPA：容易实现转场动画
      • MPA：无法实现转场动画
    5. 数据传递
      • SPA：容易传递数据，方法有很多，Vuex
      • MPA：传递数据不方便，一般用url,cookie，本地储存
    6. 搜索引擎优化（SEO）
      • SPA：需要单独方案，实现比较困难
      • MPA：实现方法容易
    7. 使用范围
      • SPA：高要求的用户体验，追求页面流畅
      • MPA：适用于高度要求搜索引擎的应用
    8. 开发成本
      • SPA：长期借助框架，有风险，万一框架不维护
      • MPA：用原生的css,js 写，代码比较多
    9. 维护成本
      • SPA：比较低
      • MPA：比较高
    10. 结构
      • SPA：一个主页面 + 许多模块组件
      • MPA：许多完整的页面
    11. 资源文件
      • SPA：组件公用的资源文件只需加载一次
      • MPA：每个页面都需要单独加载自己的资源
